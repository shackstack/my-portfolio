CASE 01 · 2025 —

반복되는 업무는
워크플로우로 만듭니다.

배포·이슈 관리·보고·코드리뷰·화면 작업·도메인 문서 작업까지, 흩어져 있던 반복 절차를 독립적인 커스텀 스킬로 나누고 6개의 핵심 워크플로우로 통합했습니다.

Claude Code Custom Skills Plugin Architecture Figma · Playwright · GitHub MCP JIRA · Confluence · Bitbucket · Slack Jenkins (CI/CD)
Problem
티켓 생성·개발서버 반영·PR·배포·보고가 제각기 수작업으로 흩어져 있었습니다. 절차가 누락되거나 휴먼에러가 생기기 쉬웠고, 기준도 사람마다 달랐습니다.
Approach
LLM을 코드 생성 도구가 아니라 반복 업무를 표준화하는 실행 계층으로 활용했습니다. 각 단계를 독립 스킬로 분리하고 MCP로 도구를 연동해 하나의 워크플로우로 묶되, 되돌리기 어려운 작업은 사람의 확인 게이트로 남겼습니다.
Impact
개인 생산성을 넘어 팀 단위 워크플로우 표준화로 이어졌고, 주당 약 4시간의 반복 업무 시간을 되찾았습니다.
01

배포 파이프라인 단일 명령화

git remote를 감지해 Jenkins Job을 자동 매핑하고, 머지·tsc·PR 생성을 백그라운드 병렬로 처리합니다. 푸시 완료를 하드 배리어로 두어 항상 최신 코드 기준으로 빌드를 트리거합니다.

15분 → 2분 (85%↓)
02

운영 배포 — 승인 게이트 준수

팀의 기존 승인 게이트 프로세스에 맞춰 Slack 승인 요청 작성·전송과 JIRA 상태 전이를 자동화했습니다. 승인 여부는 JIRA 티켓 상태 변경을 감지해 확인하고, PR 머지와 최종 실행 판단은 사람이 직접 합니다.

자동화 ∩ 안전성
03

업무 보고 자동화

JIRA 티켓과 Slack 활동을 자동 수집해 Claude가 정제한 뒤 Confluence 주간보고와 데일리 스크럼에 반영되도록 구성했습니다.

40분 → 5분
04

코드리뷰 표준화

Frontend Fundamentals 4대 원칙(가독성·예측가능성·응집도·결합도)을 각각의 스킬로 분리해, 문제 유형에 맞는 일관된 리뷰 기준을 팀이 재사용할 수 있게 체계화했습니다.

리뷰 관점의 재사용
05

화면 작업 자동화

Figma MCP로 시안을 팀 컨벤션·디자인 시스템 상수에 맞춰 변환하고, 구현 화면을 Figma와 요소 단위 px로 실측 대조해 "비슷해 보인다"식 주관 판정을 프로세스에서 제거했습니다.

반복 마크업 자동화
06

약정서 생성 — 도메인 워크플로우

분기마다 docx 원문을 코드로 옮기던 수작업을 docx 파싱 → 템플릿 반영 → 백엔드 SQL 대조 → 실측 QA → 배포의 단일 흐름으로 전환했습니다. 스택이 다른 3개 레포를 하나의 플러그인 아래 레포별 스킬로 조직화했습니다.

스킬 문서에 적어둔 컨벤션은 LLM이든 사람이든 놓칠 수 있습니다. 기계적으로 판정 가능한 규칙을 정적 검사 도구 contract-lint로 옮기고 PostToolUse 훅으로 파일 수정 순간마다 자동 실행되게 했습니다. 130여 개 계약서 전체를 판정하는 첫 실행에서 수년 전 계약서에 잠재해 있던 동일 유형의 결함을 곧바로 찾아냈습니다.
